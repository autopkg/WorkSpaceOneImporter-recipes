## Overview

After successfully separating assignment functionality into `WorkSpaceOneAssigner.py` processor (task 030) and auto-generating separate `*.ws1-assigner.recipe.yaml` recipes for each app (task 031), the main `*.ws1.recipe.yaml` recipes now contain **orphaned input variables** that are no longer used by the `WorkSpaceOneImporter` processor and should be removed.

This cleanup task removes **3 assignment-related Input keys** from every `*.ws1.recipe.yaml` recipe and their corresponding **Arguments entries** passed to the `WorkSpaceOneImporter` processor.

---

## What Needs to Be Removed

### From the `Input:` section of each `*.ws1.recipe.yaml` recipe:

Remove these three keys:

1. **`WS1_SMART_GROUP_NAME`** — Used for simple V1 assignment (now handled by WorkSpaceOneAssigner)
2. **`WS1_PUSH_MODE`** — Used for push mode in simple V1 assignment (now handled by WorkSpaceOneAssigner)
3. **`ws1_app_assignments`** — Complex YAML structure for advanced V2 assignment rules (now handled by WorkSpaceOneAssigner)

### From the `Process: Arguments:` section for WorkSpaceOneImporter:

Remove these **two processor argument lines**:

- `ws1_smart_group_name: '%WS1_SMART_GROUP_NAME%'`
- `ws1_push_mode: '%WS1_PUSH_MODE%'`

**Note:** `ws1_app_assignments` was never passed to the WorkSpaceOneImporter processor, so it doesn't need to be removed from Arguments.

---

## What Should STAY in Input Section

These variables must **NOT be removed** — they are still used by WorkSpaceOneImporter:

- All `WS1_OAUTH_*` variables — OAuth 2.0 authentication
- All `WS1_API_*` variables — API authentication (username, password, token, B64encoded credentials)
- `WS1_API_URL` — API endpoint URL
- `WS1_CONSOLE_URL` — Console URL (used for generating result links in output)
- `WS1_GROUPID` — Organization Group ID

## Rationale

### Why Remove These Variables?

1. **Separation of Concerns** — Assignment is now handled by a separate processor (`WorkSpaceOneAssigner`), so the main import recipe should focus only on import.

2. **Cleaner Recipes** — Removing orphaned variables reduces confusion and makes each recipe's purpose clear:
   - `*.ws1.recipe.yaml` → **Import only** (upload package to WS1)
   - `*.ws1-assigner.recipe.yaml` → **Assign only** (assign to smart groups)
   - `*.ws1-pruner.recipe.yaml` → **Prune only** (remove old versions)

3. **Two-Workflow Support** — Enables both usage patterns:
   - **Simple workflow:** Run only `*.ws1.recipe.yaml` → upload apps to WS1, no assignment
   - **Complete workflow:** Chain `*.ws1.recipe.yaml` + `*.ws1-assigner.recipe.yaml` → upload, then assign

4. **DRY Principle** — The `-assigner` and `-pruner` recipe types already have their own Input sections. The main recipe should not duplicate them.

## How to Execute

### Option A: Manual Recipe-by-Recipe Edit

For each `*.ws1.recipe.yaml` file:

1. **In the Input section**, delete these three keys:
   - `WS1_SMART_GROUP_NAME: ...`
   - `WS1_PUSH_MODE: ...`
   - `ws1_app_assignments: ...` (entire nested structure)

2. **In the Process section** (under Arguments for WorkSpaceOneImporter), delete:
   - `ws1_smart_group_name: '%WS1_SMART_GROUP_NAME%'`
   - `ws1_push_mode: '%WS1_PUSH_MODE%'`

3. **Validate:**
   - Run `pre-commit run check-autopkg-recipes --files <recipe>`
   - Verify all auth variables remain intact
   - Check no orphaned `%WS1_*%` references exist

### Option B: Scripted Batch Modification (Recommended)

Create a Python script (`cleanup_recipe_input_vars.py`) to:

1. Glob-find all `ws1-recipes/*.ws1.recipe.yaml` files
2. Parse each with `yaml` module
3. **From Input dict**, remove:
   - `WS1_SMART_GROUP_NAME`
   - `WS1_PUSH_MODE`
   - `ws1_app_assignments`
4. **From Process[0].Arguments**, remove:
   - `ws1_smart_group_name`
   - `ws1_push_mode`
5. Write back and validate with `pre-commit run check-autopkg-recipes --files <recipe>`
