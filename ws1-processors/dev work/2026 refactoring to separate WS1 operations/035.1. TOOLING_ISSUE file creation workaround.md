# Tooling Issue Report: File Creation with Spaces in Path

**Date:** May 27, 2026
**Reporter:** GitHub Copilot (agent session)
**Context:** While generating `035. REFACTOR_PROMPT cleanup assignment vars.md` in the `dev work/2026 refactoring to separate WS1 operations/` folder.
**Severity:** Medium — workaround exists, but wastes tool calls / context tokens and risks corrupting terminal state.

---

## Summary

When attempting to create a Markdown file inside a folder path containing **spaces** (and an intended filename containing **periods and spaces**), both the `create_file` tool and the `run_in_terminal` heredoc fallback exhibited failures. A workaround using `create_file` with an **unescaped path containing literal spaces** ultimately succeeded.

---

## Target Path

```
/Users/MAVE/dev/codeskipper/WorkSpaceOneImporter-recipes/ws1-processors/dev work/2026 refactoring to separate WS1 operations/035. REFACTOR_PROMPT remove assignment input variables from ws1.recipe.yaml.md
```

Notable characteristics:
- Two folder names contain spaces: `dev work` and `2026 refactoring to separate WS1 operations`
- Filename starts with `035. ` (digit + period + space)
- Filename contains multiple spaces and a period before `.md`

---

## Failure #1 — `create_file` with backslash-escaped path

**Call:**
```
create_file(
  filePath="/Users/MAVE/.../dev\ work/2026\ refactoring\ to\ separate\ WS1\ operations/035.\ REFACTOR_PROMPT\ remove\ assignment\ input\ variables\ from\ ws1.recipe.yaml.md",
  content=...
)
```

**Error returned:**
```
Failed to create file ...: Invalid file name: '035.\ REFACTOR_PROMPT\ remove\ assignment\ input\ variables\ from\ ws1.recipe.yaml.md'
```

**Root cause:**
The `create_file` tool treats backslash-escaped spaces (shell-style escaping) as part of the filename rather than interpreting them as escape sequences. The validator then rejects `\ ` as an invalid filename character.

**Lesson:** `create_file` expects **literal paths**, not shell-escaped paths. Do not pre-escape spaces.

---

## Failure #2 — `run_in_terminal` with heredoc

**Call:**
```sh
cat > "/Users/MAVE/.../035. REFACTOR_PROMPT remove assignment input variables from ws1.recipe.yaml.md" << 'EOF'
# 035. REFACTOR_PROMPT: ...
... (large multi-line markdown body) ...
EOF
```

**Observed behavior:**
- Terminal entered heredoc mode (`heredoc>` prompt) but **never received** the closing `EOF` marker.
- Output stream became filled with corrupted/interleaved characters showing heredoc continuation prompts interspersed with fragments of the document content.
- Subsequent `ls`, `wc -l`, and `head` commands appeared to be **swallowed inside the still-open heredoc**, never executing.
- A second attempt using a different sentinel (`ENDMARK`) also failed in the same way, this time displaying `cmdand heredoc>` (zsh continuation prompts indicating both an unclosed command **and** an unclosed heredoc).

**Likely root causes (any/all of):**
1. **Content size** — The markdown body was ~6–8 KB with many lines; the terminal tool may truncate or mishandle very large single commands sent over its stdin transport.
2. **Special characters in content** — The markdown contained backticks, `$`, `*`, emoji (`✅`, `❌`), em-dashes, and YAML fences. While `'EOF'` (single-quoted) should suppress shell expansion, **some transports may pre-process or re-encode the payload** before zsh sees it, breaking heredoc boundary detection.
3. **Sentinel collision possibility** — Unlikely (sentinel `EOF` did not appear in body), but worth noting.
4. **Persistent shell state pollution** — Once the first heredoc broke, the zsh session was left in an inconsistent state (`cmdand` prefix on prompt), poisoning subsequent commands.

**Lesson:** Avoid heredoc for **large multi-line content** via the terminal tool, especially when content includes shell-significant characters. The terminal session can be left in a broken state that affects all subsequent commands.

---

## Workaround That Succeeded

**Call:**
```
create_file(
  filePath="/Users/MAVE/dev/codeskipper/WorkSpaceOneImporter-recipes/ws1-processors/dev work/2026 refactoring to separate WS1 operations/035. REFACTOR_PROMPT cleanup assignment vars.md",
  content="<full markdown body>"
)
```

**Differences from Failure #1:**
- Path was passed with **literal spaces** (no backslash escaping).
- Filename was **shortened** (`cleanup assignment vars` instead of `remove assignment input variables from ws1.recipe.yaml`) — though this was likely not strictly necessary, it reduced risk of any length-based validation issues.

**Result:** File created successfully on first try (7.2 KB, verified via `ls -lah`).

## Recommendations

### For Agents / Operators

1. **Prefer `create_file` over terminal heredoc** for any new file with non-trivial content (> ~20 lines or containing shell-special characters).
2. **Pass paths to `create_file` literally** — do not shell-escape spaces or other characters.
3. **If `create_file` fails**, try once more with a slightly different (still descriptive) filename before falling back to terminal-based approaches.
4. **If heredoc must be used**, prefer:
   - `python -c "open('path', 'w').write('''...''')"`  for medium content, or
   - Write content to a tmp file first via multiple small `echo` appends, then `mv`.
5. **After a failed heredoc**, send a lone `EOF` (or `ENDMARK`) plus newline to close any dangling heredoc before issuing further terminal commands — otherwise the terminal session may be unrecoverable for the rest of the conversation.

### For Tool Maintainers (issues to address separately)

1. **`create_file` error message could be more helpful** — instead of `Invalid file name: '<escaped name>'`, it could hint: *"path appears to contain backslash escapes; pass the literal path instead"*.
2. **`create_file` should support paths with spaces** as first-class inputs (it does — but the error message in Failure #1 misled the agent into thinking the path itself was rejected, when really the escape sequences were the problem).
3. **`run_in_terminal` heredoc robustness** — investigate whether large heredoc payloads are being truncated, re-encoded, or losing their terminating sentinel in transit. Consider:
   - Documenting a maximum reliable size for heredoc payloads.
   - Auto-detecting an unclosed heredoc (continuation prompts in output) and auto-injecting a sentinel to recover the session.
   - Providing a dedicated "write file contents" terminal helper instead of relying on shell heredoc.
4. **Persistent shell pollution** — when a command leaves zsh in a continuation state, the terminal tool should either (a) detect this and reset, or (b) clearly surface this state to the agent so it can recover.
